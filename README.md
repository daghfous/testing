## Overview

This folder includes the User Management Backend.
It is a Python web application based on [ateme.openapi](https://gitlab.ateme.net/sw/micro-services/micro-service-store/generic-components/openapi/-/blob/master/README.md?ref_type=heads).

The latest api documentation is generated by the Gitlab CI pipeline and can be found at the following location: https://doc.ateme.net/mss/micro-service-store/pilot-foundation-services/user-management/develop/api.html

## Prerequisites

- [python3.11](https://www.python.org/downloads/) as Python interpreter,
- [pip](https://pip.pypa.io/en/stable/installing/) (already installed if you are using Python 2 >=2.7.9 or Python 3 >=3.4))
- [tox](https://tox.wiki/en/latest/install.html) based on the local [tox.ini](./tox.ini).
- [xmlsec](https://pypi.org/project/xmlsec) dependencies.
- [python-ldap](https://www.python-ldap.org/en/python-ldap-3.4.3/installing.html#debian) dependencies.

```console
sudo apt install -y pkg-config libxml2-dev libxmlsec1-dev libxmlsec1-openssl build-essential python3-dev libldap2-dev libsasl2-dev libssl-dev
```

## Python-LDAP limitations

We use the `python-ldap` library in the User-Management backend. Be aware of the following limitations and considerations:

1. **Logging and Debugging**
   - `python-ldap` relies on a C library internally, which logs debug messages differently from Python.
   - To enable debug logs, you must set the debug level via the flag:
     ```bash
        --ldap-log-detail-level=255
     ```
   - Redirecting logs to Python's standard `logging` via a filter or handler is **not supported natively**.

2. **TLS / SSL**
   - The C library requires all TLS files (key, cert, CA) to exist and be readable at runtime.

3. **Environment Variables**
   - Certain configurations rely on environment variables, e.g. `SSL_CERT_FILE`, to point to certificates.
   - These variables must be set correctly in containerized or distroless environments, otherwise TLS connections fail.

4. **Network / Timeout**
   - Network timeouts are set via the flag:
     ```bash
     --ldap-network-timeout=5
     ```

5. **Search Results**
   - `search_s()` returns a list of `(dn, entry)` tuples.
   - Attribute values can be `bytes` or `str`, so decoding is usually required:
     ```python
     if isinstance(value, bytes):
         value = value.decode(encoding='utf-8')
     ```

6. **Limitations Summary**
   - No direct Python logging integration for C-level logs.
   - TLS requires explicit handling of cert/key files.
   - Some options are only configurable via `ldap.set_option`.
   - Environment and file paths must be correct in containerized environments.

> Tip: If you are working on LDAP or modyfing code: Always test TLS connections and logging behavior in your environment before deploying to production.

## Run tests

We use tox to manage python env and requirements. So we recommend to use this tool to run tests. There is some
example of commands:

```bash
# Run with verbose and capture log
tox -- -vvs tests

# Run only one test file
tox -- tests/test_login.py

# Stop at first test fail and pop debugger at line of failling test
tox -- tests -x --pdb

# You may want to run tests which concerns only a specific datastructure
tox -- tests -m user

# You can retrieve available markers with the following command:
tox -- tests/ --markers
```

We define custom pytest argument, there the list of argument and purpose:

* `--keep-containers` Keep database container after tests suite finish. Default: False
* `--remote-database-url=DATABASE_URL` Database remote url. If set, no database container will be created at setup. Default: None
* `--database-name=DATABASE_NAME` Database remote name. Default: users
* `--wait-database-timeout=WAIT_DATABASE_TIMEOUT` Wait database replicaset init in seconds. Default 30

There are some tests that need CI vars, we can set these env var to run locally:

```bash
export ATEME_HUB_RELEASE_USR=<username>
export ATEME_HUB_RELEASE_PWD=<password>
```

## Write tests

In order to write effective and usefull test, we develop some fixture to help user to write "unit" test.
There is an overview and basic usage of fixture available:

* `init_database` This fixture with "function" scope instantiates a `Database` class and calls `initialize` at setup if behavior not override by parametrize. And finally at teardown drops the database. This fixture allows to guarantee that inserted data will be clean for next test.

```python
@pytest.mark.parametrize("scope, exception_expected, result_expected", [
    (
        {"id": "operator"},
        pytest.raises(SomeException()),
        {"acknowledged": False}
    )
])
def test_create_scope(init_database, scope, exception_expected, result_expected):
    _db = init_database
    with execption_expected:
        res = await _db.create_scope(scope)
        assert res == result_expected
```

* `populate_data`, this fixture have to be used with parametrize indirect. This will populate data configure on test. Parameter have to be
    a dict, with key as collection name and value a list of data pass to insert_many operation.
* `settings`, This fixture help to configure UserManagementApi, as this fixture is define with a function scope every changes made to settings instance during a test will be clean at teardown.
* `init_api`, Will create an instance UserManagementApi with database init by `init_database` fixture.
* `multiple_api`, this fixture have been added to cover replica use case.
* `init_backend`, this fixture inherit init_api and aiohttp_client fixture and will create `OpenApiWebApplication`, will run api and initialize TestClient, thanks to pytest-aiohttp.
    This fixture can take parameter in order to mock initialize function of UserManagementApi, this can be usefull to avoid run every task of UserManagementApi.

```python
@pytest.mark.parametrize("init_backend", [{"mock_initialize": True, "db_initialize": True}], indirect=True)
async def test_create_admin(init_backend):
    _api, _ = init_backend
    resp = await _api.post("/admin", json={"username": "admin", "password": "Admin!A0"})
    assert resp.status == 200
    body = await resp.json()
```
* `init_backend_with_admin`, inherit `init_backend` fixture and will create admin user with `admin_user` fixture credential. Will return aiohttp client, UserManagementApi instance and token freshly created.

```python
async def test_api_validate_idp_config(init_backend_with_admin, mocker):
    _api, _, _admin_token = init_backend_with_admin
    _admin_headers = {"Authorization": f"Bearer {_admin_token}"}

    username = "john_doe"
    password = "password123"

    mocker.patch("ateme.um_backend.ldap.LdapClient.bind", return_value=(None, True))

    # Test ipdconfig/validate route
    resp = await _api.post(
        "/idpconfigs/validate",
        json={**IDP_LDAP_CONFIG.to_dict(), "username": username, "password": password},
        headers=_admin_headers,
    )
    assert resp.status == 200
    assert (await resp.text()) == 'Ok - valid LDAP config', 'This LDAP config should be valid'
```

* `create_users`, this fixture will create use configure via parametrize indirect.

```python
@pytest.mark.parametrize(
    "create_users",
    [
        ([{"username": "Pavard", "password": "SecondPoteau!0"}]),
    ],
    indirect=["create_users"],
)
async def test_delete_token(init_backend_with_admin, create_users):
    """

    Login with the same user two times.
    Delete the token should logout both user connections
    """
    _api, _, admin_token = init_backend_with_admin
    _username = create_users[0]["username"]
    _headers = {"Authorization": f"Bearer {admin_token}"}

    # First login without the idp_name (default value DEFAULT_LOCAL_IDP_NAME)
    first_login = await _api.post("/token", json=create_users[0])
    assert first_login.status == 206
```

* `create_and_log_users`, Same as previous fixture but will log users too and yield tokens.

## Build artefacts

### Python Package

> **NOTE**: this procedure has been automated with the command `PYTHON=<python path> make backend-dist`

1. Install [twine](https://github.com/pypa/twine/) which is an utility for publishing Python packages on PyPi and [build](https://pypi.org/project/build/) which is now required.
```bash
python3 -m pip install twine
python3 -m pip install build
```

2. Build the package:
```bash
python3 -m build . -o dist
```

3. Push the package archive on PyPi:
```bash
python3 -m twine upload --repository-url https://nexus-rennes.ateme.net/repository/pypi-dev-hosted/ -u <username> -p <password> dist/<package>.tar.gz
```

4. Go to [Nexus](https://nexus-rennes.ateme.net/#browse/browse:pypi-dev) to verify your package has been uploaded.

### Docker image

You can use [Makefile](../Makefile) target `backend-image` in order to build backend docker image.
If you need to add some custom args, there is an args for this purpose `BUILD_EXTRAFLAGS`, for example:

```bash
BUILD_EXTRAFLAGS="--progress=plain --no-cache --pull" make backend-image
```

This will append this extra flags to the docker build command.
If you need to tag your image with a specific tag you can run:

```bash
TAG=$USER make backend-image
```

List of arguments you can override to build your image:

* `PYTHON`: Python path bin.
* `DOCKER`: Docker path bin.
* `REGISTRY`: Docker registry, default: nexus-rennes.ateme.net:10443.
* `PYTHON_DISTROLESS_BASE_IMAGE`: Image use to build the final image, default content of PYTHON_DISTROLESS_BASE_IMAGE file.
* `PYTHON_COMPILER_IMAGE`: Image use to build python packages, default content of `PYTHON_COMPILER_IMAGE` file.
* `TAG`: Docker images tag, default result of a git describe command.
* `BUILD_EXTRAFLAGS`: Extra flags you can pass to the docker build, default empty.
* `PYTHON_TAG`: Version of the python installed, by default if we find `setuptools_scm` we are trying to determine the current version else we fallback with a git describe tag.

### default-auth-data-path

> **NOTE:**
> `--default-auth-data-path <file>` can be repeated to pass multiple files in the command line.
> If this parameter is set, the parameters `--default-actions-path`, `--default-scopes-path` and `--default-descriptor-path` would not be taken into account.
> If an api descriptor has actions and scopes fields, these fields will be ignored.

We recommend to generate this file with the [generate-auth script](../backend/ateme/um_backend/scripts/generate_auth/README.md).

# auth_data.json
{
  "api_descriptors": [
    ...
  ],
  "actions": [
    ...
  ],
  "scopes": [
    ...
  ]
}
```

export PYTHONPATH=.
python -m ateme/um_backend/__main__.py --um-api-port 2529 --um-database-url <database_url> --default-auth-data-path auth_data.json.json
```
